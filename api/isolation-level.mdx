# IsolationLevel

Configures the isolation level for database transactions, determining the conflict detection strategy and visibility semantics.

## Overview

SlateDB supports two isolation levels that provide different trade-offs between performance and serializability guarantees:

- **Snapshot Isolation (SI)** - Default level, detects write-write conflicts only
- **Serializable Snapshot Isolation (SSI)** - Stricter level, detects both read-write and write-write conflicts

## Enum Variants

### Snapshot

Provides snapshot isolation with write-write conflict detection.

<ParamField path="Snapshot" type="IsolationLevel">
  Default isolation level that prevents write-write conflicts
</ParamField>

**Characteristics**:
- **Conflict Detection**: Write-write conflicts only
- **Performance**: Higher throughput due to minimal tracking overhead
- **Guarantees**: Prevents lost updates but allows write skew anomalies
- **Read Tracking**: Reads are not tracked by default (unless `mark_read()` is used)
- **Use Cases**: High-concurrency workloads where write skew is acceptable

**Example**:
```rust
use slatedb::{Db, IsolationLevel};

let db = Db::open("path/to/db", object_store).await?;
let txn = db.begin(IsolationLevel::Snapshot).await?;

txn.put(b"key", b"value")?;
txn.commit().await?;
```

### SerializableSnapshot

Provides serializable snapshot isolation with full conflict detection.

<ParamField path="SerializableSnapshot" type="IsolationLevel">
  Stricter isolation level that prevents both read-write and write-write conflicts
</ParamField>

**Characteristics**:
- **Conflict Detection**: Both read-write and write-write conflicts
- **Performance**: Lower throughput due to read tracking overhead
- **Guarantees**: Prevents write skew and provides true serializability
- **Read Tracking**: All reads are automatically tracked for conflict detection
- **Range Tracking**: Scans track read ranges to detect phantom reads
- **Use Cases**: Critical operations requiring full serializability

**Example**:
```rust
use slatedb::{Db, IsolationLevel};

let db = Db::open("path/to/db", object_store).await?;
let txn = db.begin(IsolationLevel::SerializableSnapshot).await?;

txn.put(b"key", b"value")?;
txn.commit().await?;
```

## Conflict Detection Behavior

### Write-Write Conflicts

Both isolation levels detect write-write conflicts:

```rust
let db = Db::open("path/to/db", object_store).await?;
db.put(b"balance", b"100").await?;

// Transaction 1
let txn1 = db.begin(IsolationLevel::Snapshot).await?;
txn1.put(b"balance", b"150")?;

// Transaction 2
let txn2 = db.begin(IsolationLevel::Snapshot).await?;
txn2.put(b"balance", b"200")?;

// First commit succeeds
txn1.commit().await?;

// Second commit fails with TransactionConflict
assert!(txn2.commit().await.is_err());
```

### Read-Write Conflicts (SSI Only)

Only SerializableSnapshot detects read-write conflicts:

```rust
let db = Db::open("path/to/db", object_store).await?;
db.put(b"key1", b"value1").await?;
db.put(b"key2", b"value2").await?;

// Transaction 1 reads and writes
let txn1 = db.begin(IsolationLevel::SerializableSnapshot).await?;
let _val = txn1.get(b"key2").await?;  // Read tracked automatically
txn1.put(b"key1", b"new_value1")?;

// Transaction 2 writes to key2
db.put(b"key2", b"new_value2").await?;

// Transaction 1 commit detects read-write conflict on key2
assert!(txn1.commit().await.is_err());
```

With Snapshot isolation, the above transaction would succeed:

```rust
let txn1 = db.begin(IsolationLevel::Snapshot).await?;
let _val = txn1.get(b"key2").await?;  // Read NOT tracked
txn1.put(b"key1", b"new_value1")?;

// Concurrent write to key2
db.put(b"key2", b"new_value2").await?;

// Commit succeeds (no read tracking)
assert!(txn1.commit().await.is_ok());
```

### Phantom Read Detection (SSI Only)

SerializableSnapshot tracks scanned ranges to detect phantom reads:

```rust
let db = Db::open("path/to/db", object_store).await?;
db.put(b"key1", b"v1").await?;
db.put(b"key3", b"v3").await?;

// Transaction 1 scans a range
let txn1 = db.begin(IsolationLevel::SerializableSnapshot).await?;
let mut iter = txn1.scan(b"key1"..=b"key5").await?;
while let Some(_) = iter.next().await? {
    // Range is tracked
}
drop(iter);

// Transaction 2 inserts into the scanned range
db.put(b"key2", b"v2").await?;

txn1.put(b"other", b"value")?;

// Transaction 1 commit detects phantom read conflict
assert!(txn1.commit().await.is_err());
```

## Isolation Level Comparison

| Feature | Snapshot | SerializableSnapshot |
|---------|----------|---------------------|
| Write-Write Conflicts | ✅ Detected | ✅ Detected |
| Read-Write Conflicts | ❌ Not Detected | ✅ Detected |
| Phantom Reads | ❌ Not Detected | ✅ Detected |
| Read Tracking Overhead | ❌ None | ✅ Automatic |
| Range Tracking | ❌ None | ✅ Automatic |
| Performance | 🚀 Higher | 📊 Lower |
| Serializability | ❌ No (Write Skew Possible) | ✅ Yes |
| Default Level | ✅ Yes | ❌ No |

## Choosing an Isolation Level

### Use Snapshot Isolation When:

- **High concurrency** is required
- **Write-write conflicts** are the primary concern
- **Write skew anomalies** are acceptable for your use case
- **Read-heavy workloads** benefit from minimal overhead
- **Independent key updates** that don't require cross-key consistency

**Examples**:
- Independent counter updates
- Session storage
- Cache operations
- Event logging

### Use Serializable Snapshot Isolation When:

- **Full serializability** is required
- **Cross-key consistency** must be maintained
- **Write skew** would cause correctness issues
- **Range queries** need protection from phantoms
- **Financial operations** or other critical transactions

**Examples**:
- Bank account transfers
- Inventory management
- Constraint checking across multiple keys
- Allocating unique resources

## Advanced Patterns

### Selective Read Tracking in SI

Use `mark_read()` to enable conflict detection for specific keys in Snapshot Isolation:

```rust
let txn = db.begin(IsolationLevel::Snapshot).await?;

// Read without tracking
let _cached_value = txn.get(b"cache_key").await?;

// Read with explicit tracking
let critical_value = txn.get(b"critical_key").await?;
txn.mark_read([b"critical_key"])?;

// Commit will detect conflicts on critical_key only
txn.commit().await?;
```

### Optimistic Operations with unmark_write()

Exclude specific keys from conflict detection for optimistic updates:

```rust
let txn = db.begin(IsolationLevel::SerializableSnapshot).await?;

// Critical write (tracked for conflicts)
txn.put(b"user_balance", b"100")?;

// Optimistic write (no conflict checking)
txn.put(b"view_count", b"1000")?;
txn.unmark_write([b"view_count"])?;

txn.commit().await?;
```

### Hybrid Approach

Use different isolation levels for different transaction types:

```rust
// Read-heavy analytics query
let read_txn = db.begin(IsolationLevel::Snapshot).await?;
let mut results = vec![];
let mut iter = read_txn.scan(..).await?;
while let Some(kv) = iter.next().await? {
    results.push(kv);
}
read_txn.commit().await?;

// Critical write operation
let write_txn = db.begin(IsolationLevel::SerializableSnapshot).await?;
let balance = write_txn.get(b"balance").await?;
write_txn.put(b"balance", compute_new_balance(balance))?;
write_txn.commit().await?;
```

## Transaction Anomalies

### Write Skew (Prevented by SSI)

Snapshot Isolation allows write skew, but SSI prevents it:

```rust
// Scenario: Maintain constraint that sum of two accounts >= 0
let db = Db::open("path/to/db", object_store).await?;
db.put(b"account_a", b"100").await?;
db.put(b"account_b", b"100").await?;

// With Snapshot Isolation (write skew possible)
let txn1 = db.begin(IsolationLevel::Snapshot).await?;
let txn2 = db.begin(IsolationLevel::Snapshot).await?;

let a1 = get_balance(&txn1, b"account_a").await?;
let b1 = get_balance(&txn1, b"account_b").await?;
// Check: a1 + b1 >= 100 (true: 200)
txn1.put(b"account_a", b"-50")?;  // Withdraw 150

let a2 = get_balance(&txn2, b"account_a").await?;
let b2 = get_balance(&txn2, b"account_b").await?;
// Check: a2 + b2 >= 100 (true: 200)
txn2.put(b"account_b", b"-50")?;  // Withdraw 150

txn1.commit().await?;  // Succeeds
txn2.commit().await?;  // Succeeds - WRITE SKEW!
// Result: sum = -50 - 50 = -100 (constraint violated)

// With Serializable Snapshot Isolation (prevented)
let txn1 = db.begin(IsolationLevel::SerializableSnapshot).await?;
let txn2 = db.begin(IsolationLevel::SerializableSnapshot).await?;

// Same operations...
txn1.commit().await?;  // Succeeds
txn2.commit().await?;  // FAILS - conflict detected
```

### Lost Update (Prevented by Both)

Both isolation levels prevent lost updates:

```rust
// Transaction 1
let txn1 = db.begin(IsolationLevel::Snapshot).await?;
txn1.put(b"counter", b"10")?;

// Transaction 2
let txn2 = db.begin(IsolationLevel::Snapshot).await?;
txn2.put(b"counter", b"20")?;

txn1.commit().await?;  // Succeeds
txn2.commit().await?;  // FAILS - write-write conflict
```

## Performance Considerations

### Memory Overhead

```rust
// Snapshot Isolation: Minimal memory
let txn = db.begin(IsolationLevel::Snapshot).await?;
// Only tracks written keys

// Serializable Snapshot: Higher memory usage
let txn = db.begin(IsolationLevel::SerializableSnapshot).await?;
// Tracks written keys + read keys + read ranges
```

### Contention Impact

- **SI**: Better performance under high contention on different keys
- **SSI**: May experience more conflicts under high contention due to read tracking

### Throughput

```rust
// For read-heavy workloads
// SI: ~90-95% throughput of non-transactional reads
// SSI: ~70-80% throughput due to read tracking

// For write-heavy workloads
// SI: ~85-90% throughput
// SSI: ~75-85% throughput due to conflict checking overhead
```
